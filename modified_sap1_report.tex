\begin{abstract}
This mini-project presents a modified SAP-1 CPU designed in Verilog HDL. The processor keeps the simple structure of the original SAP-1 model while extending it with additional arithmetic instructions. It uses an 8-bit datapath, 4-bit addresses, 16 memory locations, and an 8-bit instruction format made from a 4-bit opcode and a 4-bit operand. The supported instructions are LDA, ADD, SUB, MUL, DIV, OUT, and HLT. Multiplication and division are implemented manually without using the Verilog \texttt{*} and \texttt{/} operators. Functional simulation confirms that the CPU executes the main program correctly and handles arithmetic edge cases, division by zero, and halt behavior as expected.
\end{abstract}

\section{System Architecture and Datapath}

The CPU follows a multi-cycle SAP-1 style architecture. Instead of completing an instruction in one clock cycle, the processor breaks each instruction into smaller fetch, decode, and execute steps. This makes the control flow easier to observe and matches the educational purpose of the SAP-1 computer. The datapath uses a multiplexer-based 8-bit shared bus rather than an internal tri-state bus, which makes the implementation more suitable for FPGA synthesis.

\section{Module Descriptions}

This section explains each Verilog module and the role it plays in the SAP-1 datapath. The modules are intentionally kept small so that the flow of data through the CPU can be followed clearly.

\subsection{Program Counter}

The Program Counter in \texttt{program\_counter.v} is a 4-bit register that points to the next instruction in memory. During \texttt{FETCH\_ADDR}, its value is placed on the bus and loaded into the MAR. During \texttt{FETCH\_INST}, the control unit asserts \texttt{pc\_inc}, allowing the PC to move to the next instruction address. The module also includes reset and load support, although normal execution mainly uses reset and increment.

\begin{lstlisting}[style=verilogstyle,caption={Program Counter update logic}]
if (reset)
    pc <= 4'h0;
else if (load)
    pc <= data_in;
else if (inc)
    pc <= pc + 4'h1;
\end{lstlisting}

\subsection{Memory Address Register}

The Memory Address Register in \texttt{memory\_address\_register.v} stores the 4-bit address currently sent to RAM. During instruction fetch, it receives the PC value. During memory-based instructions such as LDA, ADD, SUB, MUL, and DIV, it receives the operand field from the instruction register. In this way, the MAR acts as the address bridge between the control sequence and the RAM.

\begin{lstlisting}[style=verilogstyle,caption={MAR load logic}]
if (reset)
    address <= 4'h0;
else if (load)
    address <= data_in;
\end{lstlisting}

\subsection{RAM}

The RAM in \texttt{ram\_16x8.v} contains 16 memory locations, each 8 bits wide. It stores both program instructions and data values, following the simple memory model used by SAP-1. The CPU reads RAM through the current MAR address. Before execution begins, the testbench loads instructions and data through the programming interface \texttt{prog\_we}, \texttt{prog\_addr}, and \texttt{prog\_data}.

\begin{lstlisting}[style=verilogstyle,caption={RAM storage and program loading}]
reg [7:0] memory [0:15];
assign data_out = memory[address];

always @(posedge clk) begin
    if (prog_we)
        memory[prog_addr] <= prog_data;
end
\end{lstlisting}

\subsection{Instruction Register}

The Instruction Register in \texttt{instruction\_register.v} stores the 8-bit instruction fetched from RAM. It separates the instruction into two fields: \texttt{instruction[7:4]} as the opcode and \texttt{instruction[3:0]} as the operand. The opcode is sent to the control unit, while the operand is used as a RAM address for memory-reference instructions.

\begin{lstlisting}[style=verilogstyle,caption={Instruction Register field extraction}]
if (reset)
    instruction <= 8'h00;
else if (load)
    instruction <= data_in;

assign opcode = instruction[7:4];
assign operand = instruction[3:0];
\end{lstlisting}

\subsection{Accumulator}

The Accumulator in \texttt{accumulator.v} is the main 8-bit working register of the CPU. LDA loads RAM data directly into the accumulator. Arithmetic instructions use the accumulator as the first ALU input, and the ALU result is written back into the accumulator. Thus, most data processing operations update this register.

\begin{lstlisting}[style=verilogstyle,caption={Accumulator register logic}]
if (reset)
    data_out <= 8'h00;
else if (load)
    data_out <= data_in;
\end{lstlisting}

\subsection{B Register}

The B Register in \texttt{b\_register.v} stores the second ALU operand. For ADD, SUB, MUL, and DIV, the CPU first loads \texttt{RAM[operand]} into the B register during \texttt{ALU\_READ}. In the next state, the ALU operates on the accumulator and B register values.

\begin{lstlisting}[style=verilogstyle,caption={B Register load logic}]
if (reset)
    data_out <= 8'h00;
else if (load)
    data_out <= data_in;
\end{lstlisting}

\subsection{ALU}

The ALU in \texttt{alu.v} performs ADD, SUB, MUL, and DIV operations based on the opcode. ADD and SUB use standard 8-bit arithmetic. MUL is implemented using shift-and-add logic, and DIV is implemented using restoring division based on shift/subtract operations. Division by zero is explicitly handled by returning \texttt{8'hFF} and asserting \texttt{div\_zero}.

\begin{lstlisting}[style=verilogstyle,caption={ALU operation selection}]
case (opcode)
    OP_ADD: result = a + b;
    OP_SUB: result = a - b;
    OP_MUL: result = multiply_shift_add(a, b);
    OP_DIV: begin
        if (b == 8'h00) begin
            result = 8'hFF;
            div_zero = 1'b1;
        end else begin
            result = divide_restoring(a, b);
        end
    end
    default: result = a;
endcase
\end{lstlisting}

\subsection{Output Register}

The Output Register in \texttt{output\_register.v} stores the value produced by the OUT instruction. When OUT executes, the control unit selects the accumulator onto the bus and asserts \texttt{out\_load}. The output register then holds this value as the external CPU output.

\begin{lstlisting}[style=verilogstyle,caption={Output Register load logic}]
if (reset)
    data_out <= 8'h00;
else if (load)
    data_out <= data_in;
\end{lstlisting}

\subsection{Control Unit}

The Control Unit in \texttt{control\_unit.v} is an FSM that coordinates all CPU operations. It generates the bus select signal, register load enables, PC increment signal, ALU opcode, and halted output. Its states implement instruction fetch, opcode decode, memory address setup, data read, ALU write-back, output write, and halt.

\begin{lstlisting}[style=verilogstyle,caption={Control Unit state register and default controls}]
always @(posedge clk or posedge reset) begin
    if (reset)
        state <= FETCH_ADDR;
    else
        state <= next_state;
end

bus_sel = BUS_ZERO;
mar_load = 1'b0;
ir_load = 1'b0;
pc_inc = 1'b0;
a_load = 1'b0;
b_load = 1'b0;
out_load = 1'b0;
\end{lstlisting}

\subsection{Bus Multiplexer}

The Bus Multiplexer in \texttt{bus\_mux.v} implements the shared 8-bit datapath. It selects one source at a time from PC, RAM data, IR operand, accumulator data, ALU result, or zero. The PC and operand are only 4 bits, so they are zero-extended to 8 bits before being placed on the bus.

\begin{lstlisting}[style=verilogstyle,caption={Bus source selection from bus\_mux.v}]
case (sel)
    BUS_PC:      bus_data = {4'h0, pc};
    BUS_RAM:     bus_data = ram_data;
    BUS_OPERAND: bus_data = {4'h0, ir_operand};
    BUS_A:       bus_data = a_data;
    BUS_ALU:     bus_data = alu_result;
    default:     bus_data = 8'h00;
endcase
\end{lstlisting}

\section{Instruction Set and Opcode Format}

Each instruction is 8 bits:

\[
\text{Instruction} = \{\text{opcode}[3:0], \text{operand}[3:0]\}
\]

\begin{center}
\begin{tabular}{>{\ttfamily}c l l}
\toprule
Opcode & Instruction & Operation \\
\midrule
1 & LDA addr & A = RAM[addr] \\
2 & ADD addr & A = A + RAM[addr] \\
3 & SUB addr & A = A - RAM[addr] \\
4 & MUL addr & A = A * RAM[addr] using shift-add logic \\
5 & DIV addr & A = A / RAM[addr] using shift/subtract division \\
E & OUT & Output = A \\
F & HLT & Stop CPU execution \\
0 & NOP & No operation/reserved \\
\bottomrule
\end{tabular}
\end{center}

For example, \texttt{8'h4B} is interpreted as \texttt{MUL RAM[0xB]}.

\section{ALU Design}

The ALU in \texttt{alu.v} is a combinational module. It receives the accumulator value \texttt{a}, the B register value \texttt{b}, and the current opcode. From these inputs, it continuously produces an 8-bit \texttt{result} and a \texttt{div\_zero} flag. The result is only stored back into the accumulator when the control unit reaches the \texttt{ALU\_WRITE} state.

\begin{lstlisting}[style=verilogstyle,caption={ALU module interface}]
module alu(
    input [7:0] a,
    input [7:0] b,
    input [3:0] opcode,
    output reg [7:0] result,
    output reg div_zero
);
\end{lstlisting}

\subsection{ADD and SUB}

ADD and SUB use standard 8-bit binary arithmetic. ADD computes \texttt{A + B}, while SUB computes \texttt{A - B}. Because the datapath is 8 bits wide, overflow and underflow wrap around naturally. For example, \texttt{250 + 10} becomes \texttt{4}, and \texttt{3 - 10} becomes \texttt{249}.

\begin{lstlisting}[style=verilogstyle,caption={ADD and SUB selection}]
case (opcode)
    OP_ADD:
        result = a + b;
    OP_SUB:
        result = a - b;
endcase
\end{lstlisting}

\subsection{Multiplication}

MUL uses the shift-and-add multiplication algorithm. The multiplier is checked bit by bit. If a multiplier bit is 1, the current shifted multiplicand is added to the product. The multiplicand is shifted left every iteration. This avoids the Verilog \texttt{*} operator while still describing real multiplication hardware. The internal product is 16 bits, but only the lower 8 bits are returned to match the CPU datapath.

The algorithm begins by clearing the product register. It then scans the multiplier from bit 0 to bit 7. For each bit, the shifted multiplicand is added only when that multiplier bit is high. After each iteration, the multiplicand is shifted left by one position. This is equivalent to binary long multiplication and is suitable for this project because the 8-bit operand width gives a fixed loop count of eight iterations.

\begin{figure}[H]
\centering
\includegraphics[width=0.82\linewidth]{img/4-bit-multiplication-by-shift-method.jpeg}
\caption{Shift-and-add multiplication method illustrated with a 4-bit example}
\label{fig:mul_algorithm}
\end{figure}

Although Figure~\ref{fig:mul_algorithm} illustrates the method using a smaller 4-bit example, the implemented Verilog function applies the same shift-and-add process to 8-bit operands using eight fixed iterations.

\begin{lstlisting}[style=verilogstyle,caption={Shift-and-add multiplication from alu.v}]
product = 16'h0000;
shifted_multiplicand = {8'h00, multiplicand};

for (i = 0; i < 8; i = i + 1) begin
    if (multiplier[i])
        product = product + shifted_multiplicand;
    shifted_multiplicand = shifted_multiplicand << 1;
end

multiply_shift_add = product[7:0];
\end{lstlisting}

\subsection{Division}

DIV uses the restoring division algorithm. The quotient is built one bit at a time from the most significant bit to the least significant bit. At each step, the partial remainder is shifted left and compared with the divisor. If the partial remainder is large enough, the divisor is subtracted and the quotient bit is set to 1. This avoids the Verilog \texttt{/} operator and produces integer division, so any remainder is discarded.

The algorithm first initializes the quotient and remainder to zero. It then processes the dividend from bit 7 down to bit 0. At every iteration, the next dividend bit is shifted into the partial remainder. If the partial remainder is greater than or equal to the divisor, the divisor is subtracted and the corresponding quotient bit is set. Otherwise, the quotient bit remains zero. This produces an integer quotient and discards the final remainder.

\begin{figure}[H]
\centering
\includegraphics[width=0.82\linewidth]{img/Restoring Division.jpg}
\caption{Restoring division algorithm used for DIV}
\label{fig:div_algorithm}
\end{figure}

\begin{lstlisting}[style=verilogstyle,caption={Restoring division from alu.v}]
for (i = 7; i >= 0; i = i - 1) begin
    remainder = {remainder[7:0], dividend[i]};
    if (remainder >= {1'b0, divisor}) begin
        remainder = remainder - {1'b0, divisor};
        quotient[i] = 1'b1;
    end
end
\end{lstlisting}

\subsection{Division by Zero}

If the divisor is zero, the ALU returns \texttt{8'hFF} and asserts \texttt{div\_zero}:

\begin{lstlisting}[style=verilogstyle,caption={Division-by-zero handling}]
if (b == 8'h00) begin
    result = 8'hFF;
    div_zero = 1'b1;
end else begin
    result = divide_restoring(a, b);
end
\end{lstlisting}

The raw \texttt{div\_zero} signal is generated by the combinational ALU, but the CPU-level flag is latched in \texttt{sap1\_top.v}. This is done because the ALU output can change whenever its inputs change, while the CPU should only record a divide-by-zero event when a DIV result is actually written back to the accumulator during \texttt{ALU\_WRITE}. Keeping this latch in the top-level datapath also keeps the control unit focused on sequencing states instead of storing datapath error flags.

\subsection{Opcode-Based Result Selection}

The final ALU output is selected using the opcode. For unsupported opcodes, the ALU passes the accumulator value through unchanged.

\begin{lstlisting}[style=verilogstyle,caption={Complete ALU operation selection}]
case (opcode)
    OP_ADD: result = a + b;
    OP_SUB: result = a - b;
    OP_MUL: result = multiply_shift_add(a, b);
    OP_DIV: begin
        if (b == 8'h00) begin
            result = 8'hFF;
            div_zero = 1'b1;
        end else begin
            result = divide_restoring(a, b);
        end
    end
    default: result = a;
endcase
\end{lstlisting}

\section{Simulation Results}

\subsection{Main Program Test}

The testbench loads this program into RAM:

\begin{lstlisting}[style=verilogstyle,caption={Main program loaded in sap1\_tb.v}]
write_memory(4'h0, 8'h18); // LDA RAM[0x8]
write_memory(4'h1, 8'h29); // ADD RAM[0x9]
write_memory(4'h2, 8'h3A); // SUB RAM[0xA]
write_memory(4'h3, 8'h4B); // MUL RAM[0xB]
write_memory(4'h4, 8'h5C); // DIV RAM[0xC]
write_memory(4'h5, 8'hE0); // OUT
write_memory(4'h6, 8'hF0); // HLT
\end{lstlisting}

The data values are RAM[8] = 10, RAM[9] = 3, RAM[A] = 4, RAM[B] = 2, and RAM[C] = 3. Therefore:

\[
((10 + 3 - 4) \times 2) / 3 = 6
\]

\subsection{Waveform Results}

The main waveform was captured from approximately 56 ns to 130 ns. This interval shows the complete execution sequence of the main program: LDA, ADD, SUB, MUL, DIV, OUT, and HLT. The waveform includes the following key signals:

\texttt{halted} & \texttt{div\_zero} & \texttt{cpu\_state} \\
\texttt{cpu\_pc} & \texttt{cpu\_mar} & \texttt{cpu\_instruction} \\
\texttt{cpu\_a} & \texttt{cpu\_b} & \texttt{cpu\_alu\_result} \\
\texttt{cpu\_bus} & & \\
\end{tabular}
\end{center}

\begin{figure}[H]
\centering
\includegraphics[width=\linewidth]{img/waveform_defaults.jpg}
\caption{Main program waveform showing LDA, ADD, SUB, MUL, DIV, OUT, and HLT}
\label{fig:main_waveform}

Figure~\ref{fig:main_waveform} shows the complete execution of the main program. After \texttt{reset} goes low, the CPU begins at address \texttt{0}. The \texttt{cpu\_pc} signal then increases step by step as the CPU fetches each instruction. The instruction values appear in this order: \texttt{18}, \texttt{29}, \texttt{3A}, \texttt{4B}, \texttt{5C}, \texttt{E0}, and \texttt{F0}. These values match the program sequence \texttt{LDA 8}, \texttt{ADD 9}, \texttt{SUB A}, \texttt{MUL B}, \texttt{DIV C}, \texttt{OUT}, and \texttt{HLT}.

The accumulator value \texttt{cpu\_a} shows the result after each operation. First, \texttt{LDA 8} loads RAM[8] = \texttt{0A}, so the accumulator becomes 10. Then \texttt{ADD 9} adds RAM[9] = \texttt{03}, producing \texttt{0D}, or 13. Next, \texttt{SUB A} subtracts RAM[A] = \texttt{04}, so the accumulator becomes \texttt{09}. After that, \texttt{MUL B} multiplies by RAM[B] = \texttt{02}, giving \texttt{12}, or 18. Finally, \texttt{DIV C} divides by RAM[C] = \texttt{03}, so the accumulator becomes \texttt{06}.

The \texttt{cpu\_bus} signal shows the value currently being transferred inside the datapath. Its value changes often because different FSM states select different bus sources, such as the PC, RAM output, instruction operand, accumulator, or ALU result. The \texttt{cpu\_alu\_result} signal also changes whenever the ALU inputs change because the ALU is combinational. However, the ALU result is stored into the accumulator only during the \texttt{ALU\_WRITE} state.

At the \texttt{OUT} instruction, the final accumulator value \texttt{06} is copied to the output register, so \texttt{out} becomes \texttt{06}. When \texttt{HLT} is reached, the \texttt{halted} signal becomes high and the CPU stops executing. The \texttt{div\_zero} signal stays low throughout this waveform because the division operand is not zero.

\section{Design Decisions, Challenges, and Conclusion}

One important design decision was how to organize the shared bus. A tri-state bus is close to the traditional SAP-1 explanation and is easy to understand conceptually because each component can appear to ``drive'' the same bus. However, internal tri-state buses are not ideal for FPGA synthesis and can create multiple-driver problems if the control signals are not handled carefully. Therefore, this project uses a bus multiplexer instead. The trade-off is that the mux version is slightly less similar to the textbook diagram, but it is clearer in Verilog, safer for synthesis, and easier to debug in simulation. Another related decision was to latch the CPU-level \texttt{div\_zero} flag in \texttt{sap1\_top.v} rather than in the control unit. Since \texttt{div\_zero} describes the result of a datapath operation, it is stored when the ALU result is committed to the accumulator, while the control unit remains responsible only for FSM sequencing.

The main challenges were learning to write Verilog correctly and translating algorithms into circuit-style logic. MUL and DIV could not use \texttt{*} and \texttt{/}, so the arithmetic algorithms had to be rewritten as fixed hardware procedures. It also took time to understand how each module, such as the PC, MAR, IR, registers, ALU, RAM, bus, and control unit, works together across multiple clock cycles. These difficulties were solved by applying the processor datapath and control concepts studied in Module 2 and by testing each part through simulation.

In conclusion, the project successfully implements a modified SAP-1 CPU with a clear multi-cycle datapath and FSM-based control unit. The CPU supports memory access, arithmetic operations, output, and halt behavior using separate Verilog modules. Simulation results confirm that the design works correctly, including the added MUL and DIV instructions and important edge cases such as division by zero. Overall, the project helped connect the theory of CPU organization with an actual hardware description implementation.
